# 货架客户 CRM 产品开发手册

版本：1.0  
状态：产品基线  
适用范围：MVP-A 本地验证版及后续 V1 后端化  
最后更新：2026-07-10

## 1. 文档治理

本手册是产品范围、领域对象、业务规则、状态机、权限和验收口径的唯一权威来源。

- `README.md` 只描述当前仓库已经实现的能力、启动方式和已知限制。
- `AGENTS.md` 只描述工程执行约束；与本手册冲突时，以本手册的产品规则为准。
- `tasks/` 和实施计划只能细化本手册，不能改变本手册中的对象关系和业务规则。
- 任何产品规则变更必须先修改本手册，再修改模型、测试和页面。
- 手册中的“必须”是验收要求；“V1”或“后续”条目不属于当前 MVP-A 完成条件。

## 2. 产品目标和版本边界

### 2.1 产品目标

货架客户 CRM 服务于便利店、超市、药店、母婴店、仓库等货架类 B2B 销售。系统以客户为关系主体，以项目/商机为销售主线，让销售人员能够从需求发现一直推进到报价、订单和回款。

主链路：

```text
客户 → 项目/商机 → 跟进/任务/文件 → 报价版本 → 客户确认 → 订单 → 应收节点 → 收款
```

### 2.2 MVP-A 本地验证版

MVP-A 必须验证以下闭环：

1. 开发登录、个人或团队工作空间初始化。
2. 客户、联系人和项目/商机的最小录入。
3. 重复客户提示和客户归属记录。
4. 跟进原文保存、AI 建议预览和人工确认。
5. 任务创建、到期、完成和来源追踪。
6. 多行报价、版本复制、发送、确认、作废和转订单。
7. 订单金额、应收节点和分次收款闭环。
8. 文件元数据及可靠业务绑定；真实文件内容上传可延后到 V1。
9. 客户/项目业务时间线。
10. 工作台指标及点击后精确下钻。
11. 手机宽度下的快速客户、跟进、任务和文件入口。

MVP-A 仍允许 localStorage、开发登录和本地规则模拟 AI，但领域关系、状态约束和验收口径必须与 V1 一致。

### 2.3 V1 后端化

V1 包括真实认证、数据库、服务端权限、真实文件存储、审计日志、数据迁移、备份恢复、真实模型接入和可安装 PWA。

### 2.4 不在 MVP-A 范围

完整公海池、复杂审批流、ERP 库存采购、成本会计、完整配送安装、完整售后、复杂 BI、原生 App、CAD 自动识别、企业微信全量同步和私有化部署不属于 MVP-A。

## 3. 角色和权限矩阵

### 3.1 角色

- 老板/负责人 `owner`：团队全部业务数据和设置。
- 管理员 `admin`：团队全部业务数据和日常配置，不可转让工作空间所有权。
- 销售成员 `sales`：本人负责或明确协作的客户、项目及其下游对象。

### 3.2 权限矩阵

| 对象/动作 | owner | admin | sales |
|---|---:|---:|---:|
| 查看团队全部客户/项目 | 是 | 是 | 否 |
| 查看本人负责/协作客户 | 是 | 是 | 是 |
| 创建客户、项目、跟进、任务 | 是 | 是 | 是 |
| 转交客户/项目负责人 | 是 | 是 | 否 |
| 创建、编辑报价草稿 | 是 | 是 | 是，仅本人项目 |
| 发送报价 | 是 | 是 | 是，仅本人项目 |
| 客户确认/作废报价 | 是 | 是 | 是，仅本人项目 |
| 将确认报价转订单 | 是 | 是 | 是，仅本人项目 |
| 修改订单成交金额 | 是 | 是 | 否 |
| 记录收款 | 是 | 是 | 是，仅本人项目 |
| 撤销收款 | 是 | 是 | 否 |
| 查看敏感金额和联系方式 | 是 | 是 | 是，仅授权范围 |
| 团队配置与成员管理 | 是 | 是 | 否 |
| 删除/归档关键记录 | 是 | 是 | 否 |

MVP-A 前端可按此矩阵展示操作，但 V1 必须在服务端依据会话中的 `team_id`、用户、角色和数据范围再次校验。

## 4. 领域词汇和对象关系

### 4.1 核心词汇

- 客户 `Customer`：购买关系主体，可以是公司、门店经营者或个人老板。
- 联系人 `Contact`：客户下具体沟通人员；一个客户可有多个联系人。
- 项目/商机 `Opportunity`：一次具体门店、仓库、装修批次或成交机会，是销售推进主线。
- 跟进 `FollowUp`：一次沟通事实，必须保留原文。
- AI 建议 `AiSuggestion`：基于跟进原文生成、尚未写入正式字段的结构化建议。
- 任务 `Task`：需要某人在某时间前完成的动作。
- 报价组 `QuotationSet`：同一项目的一轮报价上下文。
- 报价版本 `QuotationVersion`：报价组内不可混淆的 V1、V2 等版本。
- 订单 `Order`：由一个已确认报价版本生成的成交记录。
- 应收节点 `Receivable`：订单下计划收款节点，如定金、尾款。
- 收款 `Payment`：一次实际到账事实，可对应一个应收节点。
- 文件资产 `FileAsset`：原始文件本体或本地元数据。
- 文件绑定 `FileBinding`：文件与客户、项目、跟进、报价、订单、收款之间的关系。
- 操作日志 `OperationLog`：不可由普通用户修改的关键业务操作证据。

### 4.2 权威关系

```text
Team 1─N Customer
Customer 1─N Contact
Customer 1─N Opportunity
Opportunity N─N User（负责人 1 人，协作人 0..N）
Opportunity 1─N FollowUp
FollowUp 1─N AiSuggestion
Opportunity 1─N Task
Opportunity 1─N QuotationSet
QuotationSet 1─N QuotationVersion
QuotationVersion 1─N QuotationLineItem
QuotationVersion 0─1 Order
Order 1─N Receivable
Receivable 1─N Payment
FileAsset 1─N FileBinding
Customer/Opportunity/FollowUp/QuotationVersion/Order/Payment 1─N FileBinding
```

所有团队业务对象必须包含 `team_id`。下游对象必须同时保存直接父对象 ID；为了高效校验和查询，可冗余保存 `customer_id`、`opportunity_id`，但写入时必须从父对象推导，不能信任前端自由传入。

### 4.3 现有数据迁移规则

- 每个旧 `Customer` 生成一个默认 `Opportunity`，名称为“首次项目”。
- 旧跟进、任务和报价根据 `customerId` 迁移到该默认项目。
- 旧报价每条生成一个 `QuotationSet`，原报价成为该组的一个版本。
- 旧订单的应收节点和收款拆成独立对象，并补齐 `team_id`、`customer_id`、`opportunity_id`、`order_id`。
- 迁移必须幂等；同一存储版本只执行一次，失败时保留原始数据并显示可恢复错误。

## 5. 状态机和联动

### 5.1 客户状态

客户只表示关系状态：

```text
活跃 ↔ 暂停
活跃/暂停 → 已归档
```

客户不使用“报价谈判、交付安装”等项目阶段。

### 5.2 项目/商机阶段

```text
线索 → 初步沟通 → 资料/现场 → 设计/方案 → 报价谈判
→ 成交确认 → 交付/安装 → 回款/售后 → 复购/转介绍
                       └→ 流失
```

- 创建项目默认进入“线索”。
- 创建第一条有效跟进可建议进入“初步沟通”，但必须人工确认。
- 创建报价后可建议进入“报价谈判”。
- 确认报价或创建订单后进入“成交确认”。
- 全部应收结清后进入“回款/售后”。
- 用户可回退阶段，但必须填写原因并记录日志。
- “流失”是关闭状态，必须记录原因；重新激活后回到指定阶段。

### 5.3 任务状态

```text
待处理 → 已完成
待处理 → 已取消
```

“逾期”是由 `status=待处理` 且 `due_at < 当前业务日期` 计算出的视图状态，不单独存储。

任务必须使用 `source_type + source_id` 精确关联，不得依赖标题文字匹配。自动生成任务还必须保存稳定的 `automation_key`，用于防重复和自动完成。

### 5.4 报价版本状态

```text
草稿 → 已发送 → 客户确认 → 已转订单
  │        │          │
  └→ 已作废 ├→ 已过期  ├→ 已被替代
           │          └→ 复制新版本
           └→ 已作废
```

- 新建和复制报价必须先进入“草稿”。
- 只有草稿可原地编辑。
- 发送后内容锁定；修改必须复制为新版本。
- 同一报价组允许多个历史发送版本，但最多一个有效“客户确认”版本。
- 确认新版本时，同组旧确认版本变为“已被替代”。
- 已转订单版本永久锁定，只能查看或复制。
- 作废、过期、替代和确认都必须记录操作者、时间和原因。

### 5.5 订单状态

```text
待确认 → 待收定金 → 执行中 → 待尾款 → 已完成
   └────────────────────────→ 已取消
```

- MVP-A 从确认报价生成订单时可直接确认，进入“待收定金”或“待尾款”。
- 定金节点结清后进入“执行中”或“待尾款”，由是否仍有后续应收决定。
- 全部有效应收结清后进入“已完成”。
- 已完成订单不得原地改金额；调整必须新增调整记录或撤销收款。

### 5.6 应收和收款状态

应收状态：`待收款 → 部分收款 → 已收款`，另有 `已取消`。  
收款状态：`有效 → 已撤销`。

撤销收款必须填写原因，并重新计算应收与订单状态。不得物理删除有效或已撤销收款。

## 6. 金额、版本、作废和审计规则

### 6.1 金额

- 默认币种为 CNY。
- 页面显示和业务输入最多两位小数。
- V1 数据库存储使用整数分；MVP-A 可继续使用两位小数数值，但所有计算必须通过统一金额函数。
- 报价产品小计 = 数量 × 单价。
- 报价总额 = 产品金额 + 运费 + 安装费 + 设计/测量/差旅费 − 折扣。
- 折扣不得使报价总额小于或等于零。
- 订单成交金额默认等于确认报价总额。
- 修改订单成交金额必须由 owner/admin 执行，并填写调整原因。
- 有效应收节点计划金额合计必须等于订单成交金额。
- 有效收款不得超过对应应收剩余金额。

### 6.2 版本与不可变性

- 已发送和后续状态的报价内容不可原地修改。
- 已确认订单和有效收款不得静默覆盖。
- 修改关键数据必须形成新版本、调整记录或撤销记录。
- 版本号在同一报价组内从 1 连续递增。

### 6.3 删除与归档

- 未被引用的草稿可以删除。
- 客户、项目、已发送报价、订单、应收、收款、文件绑定使用归档、作废、取消或撤销，不物理删除。
- 所有关键状态变化写入操作日志。

## 7. 文件模型

### 7.1 文件资产状态

```text
临时 → 已绑定
临时 → 已过期
已绑定 → 已归档
```

- 文件上传后先创建 `FileAsset`，状态为“临时”。
- 创建至少一个有效 `FileBinding` 后变为“已绑定”。
- 临时文件默认保留 24 小时，V1 定时清理过期对象。
- 原始文件必须保留；预览图和 AI 提取结果不能替代原文件。

### 7.2 可绑定对象

文件可绑定：客户、项目、跟进、报价版本、订单和收款。一个文件可以有多个绑定，但每个绑定必须包含 `team_id`、`file_id`、`target_type`、`target_id`、上传人和创建时间。

### 7.3 收款凭证

- 记录收款时凭证在 MVP-A 为可选，在 V1 可由团队配置为必填。
- 凭证文件必须绑定到具体 `Payment`，不能只绑定订单。
- 撤销收款不删除凭证，只将绑定随业务记录保留为历史证据。

## 8. AI 建议模型

### 8.1 输出契约

AI 输出必须符合：

```json
{
  "summary": "",
  "store_type": "",
  "area": "",
  "location": "",
  "product_demand": [],
  "budget": "",
  "opening_time": "",
  "objections": [],
  "blockers": [],
  "risk_tags": [],
  "next_action": "",
  "next_follow_time": "",
  "confidence": 0,
  "evidence": []
}
```

### 8.2 生成和确认流程

```text
保存跟进原文 → 生成 AI 建议 → 展示建议与证据
→ 用户勾选字段 → 人工确认 → 写入客户/项目/任务正式字段
```

- `AiSuggestion` 必须独立保存原始输出、解析结果、模型标识、提示词版本、生成时间和状态。
- AI 失败不得阻止保存跟进原文。
- JSON 解析失败时显示纯文本，记录失败日志，不写正式字段。
- 未经人工确认，AI 只能写入建议对象，不能覆盖客户、项目、报价、订单、收款、权限或文件。
- 确认动作保存确认人、确认时间和所选字段。
- AI 查询范围继承当前用户的数据权限。

## 9. 工作台指标和下钻

所有指标与下钻列表必须调用同一查询函数和同一筛选条件。

| 指标 | 定义 | 下钻条件 |
|---|---|---|
| 今日待办 | 待处理且 `due_at` 等于今天 | 任务列表：状态待处理、日期今天 |
| 逾期任务 | 待处理且 `due_at` 早于今天 | 任务列表：状态待处理、日期早于今天 |
| 活跃项目 | 未流失、未归档、未完成 | 项目列表：活跃状态 |
| 待确认报价 | 最新有效版本状态为已发送 | 报价列表：已发送、排除过期作废 |
| 有效订单 | 非已取消订单 | 订单列表：排除已取消 |
| 未收金额 | 有效应收计划金额减有效收款 | 应收列表：剩余金额大于零 |
| 逾期应收 | 到期日前于今天且剩余金额大于零 | 应收列表：逾期、未结清 |
| 本月已收 | 本月有效收款金额合计 | 收款列表：本月、状态有效 |

“报价总额”若展示，只统计每个报价组的最新有效版本，不累计所有历史版本。

业务日期按团队时区计算；默认 `Asia/Shanghai`。时间存储统一使用 UTC ISO 8601。

## 10. Web 和移动端页面范围

### 10.1 Web 页面

1. 工作台：指标、今日任务、逾期应收和下钻。
2. 客户：列表、详情、联系人、归属历史。
3. 项目：阶段、负责人、需求、时间线。
4. 跟进：原文、AI 建议、人工确认。
5. 任务：今日、逾期、全部及来源。
6. 报价：报价组、版本、行项目、费用和状态操作。
7. 订单/回款：订单、应收节点、收款和凭证。
8. 文件：按业务对象查看绑定。
9. 团队/权限：成员、角色和数据范围。
10. 设置：行业配置、团队时区、凭证要求。

### 10.2 MVP-A 移动端

移动端只提供：今日任务、客户搜索、客户摘要、快速新增客户、快速跟进、拍照/选文件、完成任务和简单项目时间线。

报价编辑、订单金额调整、权限设置和复杂报表只在 Web 端提供。MVP-A 只要求响应式页面；可安装 PWA、离线缓存和推送属于 V1。

## 11. 非功能要求

- 团队隔离：任何业务读取和写入都由会话推导 `team_id`。
- 数据兼容：localStorage 结构升级必须有版本号、迁移器和失败回退。
- 幂等：转订单、确认报价、自动任务和收款提交必须有幂等保护。
- 审计：确认、作废、转单、金额调整、收款、撤销、转交和权限修改必须记录日志。
- 性能：MVP-A 典型 1,000 客户、10,000 时间线事件下，常用列表首次展示目标小于 2 秒。
- 文件：V1 默认单文件上限 20 MB；允许团队配置类型白名单。
- 隐私：联系方式、文件和 AI 原文不得跨团队暴露；日志不得记录密码、令牌或完整敏感内容。
- 备份：V1 每日备份；恢复目标 RPO 24 小时、RTO 4 小时。
- 可访问性：关键表单有标签、错误信息和键盘可操作性。
- 兼容性：当前 Chrome、Edge 稳定版及最近一个主要版本；移动端覆盖常见 375px 宽度。

## 12. MVP-A 验收路径

1. 使用开发账号进入并创建团队工作空间。
2. 创建客户、联系人和第一个项目。
3. 重复手机号触发明确的重复客户提示。
4. 保存跟进原文，生成 AI 建议，但正式字段在确认前不变化。
5. 人工确认部分 AI 建议，生成下一步任务。
6. 创建多行报价草稿，验证金额计算。
7. 发送报价后不能原地修改；复制生成下一版本。
8. 确认一个版本，同组不存在第二个有效确认版本。
9. 从确认报价转订单，订单金额默认等于报价总额。
10. 定金和尾款应收合计等于订单金额。
11. 分次收款、超额收款拦截和全部结清状态正确。
12. 收款凭证绑定到具体收款记录。
13. 客户/项目时间线展示跟进、任务、报价、订单、收款和文件事件。
14. 工作台每个数字点击后进入匹配列表，列表数量和金额与数字一致。
15. 销售成员无法读取其他销售未授权的项目；owner/admin 可以。
16. 手机宽度下完成客户、跟进、任务和文件的快速操作。
17. 旧 localStorage 数据迁移后仍可查看，失败时原始数据不丢失。

## 13. 实施顺序

后续开发严格按以下阶段进行，每阶段完成测试和数据兼容后再进入下一阶段：

1. 引入 `Contact`、`Opportunity` 和旧数据幂等迁移。
2. 拆分报价组/版本并落实报价状态机。
3. 拆分订单、应收和收款对象，统一金额规则。
4. 引入文件资产/绑定和收款凭证关系。
5. 引入独立 AI 建议及人工确认流程。
6. 落实角色权限矩阵和数据范围。
7. 统一工作台查询与下钻。
8. 完成移动端最小页面和跨浏览器验证。

每个阶段必须同时更新模型、存储迁移、服务、页面、自动化测试和 README 当前状态。不得只改页面或只改类型声明。
